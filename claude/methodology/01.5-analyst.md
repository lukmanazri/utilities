# Agent: Analyst — Comprehension Engine

> Responsibility: Build a complete, durable understanding of the application BEFORE any vulnerability hunting begins. No hunting until the code is understood.
> Output: projectname_master/01.5-analyst.md + comprehension/ folder
> Model: subsystem fan-out subagents → **Sonnet** (parallel breadth); synthesis + teach-back → **Opus** (cross-cutting judgment)
> Runs between Mapper (Step 1) and Hunter (Step 2). This stage is the depth the pipeline was missing. It is also a re-entrant SERVICE (see § SERVICE MODE).

---

## WHY THIS STAGE EXISTS

The old pipeline went Mapper (breadth: "where are the doors?") → Hunter (grep for sink
names). Nothing in between ever *understood* the application. So every engagement was
shallow regardless of how many tokens were spent: leads came from grep coincidences, not
from a model of how the app actually works. Bugs that require understanding — violated
ownership invariants, broken state machines, trust-boundary confusion, assumption gaps —
were structurally invisible.

**Analyst fixes this.** It spends tokens deliberately and expensively to understand the
code, and it writes that understanding to disk as durable artifacts so the spend survives
context resets. Hunter then hunts from *hypotheses grounded in understanding*, not blind
greps.

---

## THE RULE — G9 COMPREHENSION GATE (this stage enforces it)

> **No stage may file, hunt, trace, exploit, or chain a vulnerability in any subsystem whose
> comprehension doc does not exist and whose teach-back has not passed.**

Understanding is proven by a written artifact, not by a claim. "I read it" is not
comprehension. The teach-back (below) is the oracle: if the agent cannot narrate how a
request flows end-to-end through a sensitive endpoint and name every trust boundary it
crosses, it does not understand the code and may not proceed to hunt it.

Every downstream stage's first action is to read `comprehension/coverage.md`. Any subsystem
not marked `✅ Understood` is **off-limits**. If a later stage needs a subsystem that isn't
comprehended, it calls Analyst in SERVICE MODE rather than working blind — the gate is never
a dead end, but it is never bypassed either.

---

## SINGLE-WRITER COMPLIANCE (state discipline)

Analyst fans out many subagents. Per CLAUDE.md § STATE DISCIPLINE:
- Each subsystem subagent writes ONLY its own `comprehension/subsystem/<name>.md`. It never
  touches the master index, never edits `invariants.md`, never edits `coverage.md`.
- The Analyst stage agent (synthesis) is the SOLE writer that merges subagent docs into the
  shared artifacts (`invariants.md`, `domain-model.md`, …) and into `00-master-index.md §
  Comprehension Coverage`, AFTER the subagents finish.
- This is what keeps parallel comprehension from corrupting shared state.

## STEP 0 — SCOPE SELECTION (do this first; it bounds everything)

You cannot full-read an entire monorepo. Deep comprehension targets a bounded set.

**A file gets the FULL DEEP READ if it is in the intersection:**

```
(reachable from an entry point in the Attack Surface Map — including non-HTTP sources)
        ∩
(security-load-bearing: handles auth, session, access control, data access/ORM,
 file I/O, templating/rendering, deserialization, crypto, external/outbound calls,
 message queues/jobs, admin functionality, or payment/financial logic)
```

**Bias to INCLUSION (this is a standing decision).** When unsure whether a subsystem is
security-load-bearing, comprehend it. A blind spot is a *silent failure* — uncomprehended →
unhunted → invisible, the worst possible outcome. Over-reading is bounded and merely slower.
Err toward in-scope.

Everything genuinely outside the set stays at Mapper's breadth-level map — note it exists, do
not deep-read it. Record the scope decision explicitly:

```markdown
## Comprehension Scope
| Subsystem | In scope? | Reason | Entry points reaching it |
|-----------|-----------|--------|--------------------------|
| auth      | Yes       | security-load-bearing + reachable | /login, /token, ... |
| reporting | No        | reachable but no security surface | /reports |
| migrations| No        | not reachable from entry point    | — |
```

**Anti-sprawl bounds (hard):**
- Each subsystem subagent: max ~25 source files deep-read, max one agent session.
- If a subsystem exceeds that, split it (`auth-tokens`, `auth-session`, `auth-mfa`) — do
  not let one subagent run unbounded.
- If after scoping the in-scope set is still >12 subsystems, rank by (reachability ×
  privilege × input-richness) and comprehend the top 12 first; queue the rest (they remain
  off-limits to Hunter until comprehended — via this stage or SERVICE MODE).

## STEP 1 — SUBSYSTEM DECOMPOSITION

From the in-scope set, name the security-load-bearing subsystems. Typical set (adapt to
the app — do not invent subsystems that don't exist):

- `auth` — authentication: login, token issue/verify, password handling, MFA
- `session` — session/cookie lifecycle, fixation, invalidation
- `access-control` — authz: middleware, guards, role/permission checks, ownership
- `data-access` — ORM/query layer, raw query escape hatches, how queries are built
- `file` — upload, storage, retrieval, path handling, archive extraction
- `rendering` — template engines, serialization to client, auto-escape behavior
- `deserialize` — any decode of attacker-influenced structured data
- `crypto` — signing, encryption, randomness, IV/nonce handling
- `external` — outbound HTTP/SSRF surface, webhooks, third-party calls
- `jobs` — queues, consumers, background workers, scheduled tasks
- `admin` — privileged functionality, internal endpoints

For each subsystem, list its key files (file:line) from Mapper's module map.

---

## STEP 2 — FAN-OUT: ONE SUBAGENT PER SUBSYSTEM (parallel, bounded, Sonnet)

This is where the tokens get spent. Spawn one comprehension subagent **per subsystem**,
in parallel, on **Sonnet** (bounded breadth reading; many cheap readers). Each subagent:

1. Reads ONLY its subsystem's files (bounded scope = no context blowout)
2. Reads the **full implementation** of every key function — not signatures, not greps,
   the actual bodies, following calls into other in-scope files as needed
3. For library calls it cannot understand from the call site, reads the installed library
   source (`python3 -c "import x; print(x.__file__)"` / `node -e "require.resolve('x')"`)
4. Uses the **running local instance** as a truth oracle — hit the subsystem's endpoints
   through Burp, observe real parameter names, real responses, real error shapes. Source
   tells you structure; the live instance tells you actual behavior.
5. Writes `comprehension/subsystem/<name>.md` (template below) — ITS OWN FILE ONLY
6. Dies. Its understanding now lives in the doc, not in context.

**Subagent output — `comprehension/subsystem/<name>.md`:**

```markdown
# Subsystem: <name>

## Purpose
<what this subsystem is responsible for, 2-3 sentences>

## Key components
| Component | File:Line | Role |
|-----------|-----------|------|

## Control flow
<how a request/call moves through this subsystem, in execution order.
 Worked, specific, with file:line at each hop. Not a summary — a trace.>

## What it trusts
<inputs/callers/state this subsystem assumes are already validated or safe>

## What it assumes (assumption gaps)
| # | Assumption made | What is actually guaranteed | Gap |
|---|-----------------|-----------------------------|-----|
<the developer assumes X; the library/caller actually guarantees Y; the gap is the bug surface>

## Configuration
<security-relevant config: flags, env, defaults. e.g. auto-escape on/off, token expiry
 enforced, ORM raw() usage present, debug mode, CORS, cookie flags>

## Privilege & capability notes (feeds the finding schema)
<which privilege tier(s) reach this subsystem, and what capability acting here grants.
 This is the raw material Hunter uses to assign Precondition-Privilege (P) and Capability (C),
 and that the triager uses for the marginal-capability test.>

## Invariants this subsystem relies on
| # | Invariant | Relied on by | Enforced? (Y/N/Partial) | Enforced at file:line |
|---|-----------|--------------|-------------------------|------------------------|
<e.g. "order_id passed to get_order() belongs to session.user" — enforced? where?>

## Live-instance observations
<what Burp showed that source alone didn't: real param names, response differentials,
 error verbosity, headers, redirects>

## Open questions
<anything not yet understood — these block the teach-back until resolved or marked blocked>
```

If a subsystem cannot be understood (heavy obfuscation, minified vendor bundle, missing
source): mark it `⚠️ Comprehension Blocked` in coverage, document why in `Open questions`,
and add it to `00-master-index.md § Researcher Actions Required`. **Do not fake
comprehension.** A blocked subsystem is off-limits, not a free pass.

## STEP 3 — SYNTHESIS (single agent, Opus, reads the docs, not the source)

After fan-out, one synthesis pass (Opus) reads the subsystem docs (cheap — docs, not source)
and produces the cross-cutting artifacts. These are the bridge to Hunter and the triager.

### `comprehension/domain-model.md`
Entities, their fields, relationships, and **ownership/tenancy** — who owns which record,
which fields are user-settable vs server-set. This IS your IDOR/BOLA/mass-assignment
surface, derived once from understanding instead of grepped blind.

### `comprehension/request-lifecycle.md`
The middleware/filter/interceptor chain **in execution order**, with where each
cross-cutting concern fires: input parsing, authn, authz, CSRF, rate-limit,
deserialization, transaction boundaries, rendering. Then a **full worked end-to-end trace
of 3 representative endpoints**: one unauthenticated, one user-level, one admin-level.
Each trace names every hop and every trust boundary crossed. This single artifact is what
shallow mappers always miss and it's what actually determines reachability.

### `comprehension/framework-semantics.md`
The framework's actual behavior for THIS app: parameter binding (which source wins on
collision — query vs body vs path), ORM parameterization pattern and its escape hatches,
template auto-escape (on by default? bypassed anywhere?), auth-library guarantees vs. how
it's actually configured here, type-coercion quirks. Concrete, with file:line evidence.

### `comprehension/trust-boundaries.md`
Every point where data/control crosses a trust level: external→app, app→db, app→external,
service→service, user-context→admin-context, request-A-write→request-B-read (second-order).
For each: what validation exists at the boundary, and whether it's sufficient.

### `comprehension/security-model.md` — INTENDED privilege model (ground truth for G10)
> The triager (Final Boss, G10) cannot judge whether a finding crosses a real boundary
> without knowing what each privilege tier is *supposed* to be able to do BY DESIGN. Admin→RCE
> is **not a vuln** in a product where admins run code by design (e.g. plugin upload); it **is**
> a vuln in a product that markets a restricted/sandboxed admin. This file is what tells the
> triager which world it's in. Build it from docs/README/SECURITY.md AND the actual code.

```markdown
# Security Model — [projectname]

## Privilege tiers (lowest → highest)
| Tier | How you reach it | What it is DESIGNED to do (by-design capabilities) |
|------|------------------|-----------------------------------------------------|
| anonymous | no auth | browse public X, register |
| user | register/login | own-resource CRUD, ... |
| role-X | granted by admin | ... |
| admin | ... | install plugins / run code / manage users — **code-exec is intended** |

## Boundaries the product CLAIMS to enforce
<vendor-stated guarantees: tenant isolation, "users cannot read other orgs' data",
 "API tokens are scoped to project", "admin cannot read DB credentials", persistence
 survives credential rotation, etc. Cite the doc/file each comes from.>

## By-design "won't fix" classes (do NOT report as vulns at the assumed privilege)
<e.g. "admin → arbitrary file write is by design (theme editor)"; mirror upstream's
 documented security model / SECURITY.md "out of scope" list.>

## Marginal-capability reference (for the triager)
<for each high tier: what it ALREADY grants, so a vuln that merely re-grants the same
 capability at the same tier = marginal capability ~0 = not a finding.>
```

### `comprehension/invariants.md` — THE LEAD LIST (highest-yield artifact, BIDIRECTIONAL)
Consolidate every invariant from every subsystem doc into one ranked table. **Violated
invariants are the bugs.** This table is literally what Hunter consumes.

> **Bidirectional:** Analyst SEEDS this as the *initial* lead list. It is NOT final. Hunter
> ENRICHES it — Hunter's free-hunt/SAST recall pass may surface a bug with no matching
> invariant, in which case Hunter ADDS the invariant back (marked source=Hunter). Treat this
> file as living shared state, not a one-shot bottleneck. (Analyst remains the sole writer
> during this stage; Hunter becomes the writer during Step 2.)

```markdown
# Invariants — ranked lead list (bidirectional: Analyst seeds, Hunter enriches)

| # | Invariant | Domain | Relied on by | Enforced? | Enforced at | Violable? | Source | → Hunter hypothesis |
|---|-----------|--------|--------------|-----------|-------------|-----------|--------|---------------------|
| INV-001 | order_id belongs to session.user | access-control | GET /orders/{id} @ orders.py:88 | No | — | Likely | Analyst | IDOR: try another user's order_id |
| INV-002 | uploaded filename has no path separators | file | save_upload @ upload.py:42 | Partial | upload.py:50 (strips `../` once) | Maybe | Analyst | path traversal via double-encoding |
```

Rank by: (enforced? No/Partial) × (privilege of what it protects) × (input reachability).
Unenforced invariants protecting high-value resources go to the top.

### `comprehension/evolution.md`
Security-relevant history: `git log --grep` for security/CVE/fix/auth commits and read the
diffs; SECURITY.md / CHANGELOG; known CVEs for the pinned version. How the code's bugs
evolved predicts where siblings still hide. Seed Hunter and Variant Scanner with the
anti-patterns found here.

## STEP 4 — TEACH-BACK (the G9 oracle — proof of comprehension)

Comprehension is only proven when written down and complete. Produce
`comprehension/teach-back.md`:

For each of the 3 representative endpoints (unauth / user / admin), write an end-to-end
narration that:
1. Traces the request from ingress to response, naming **every** hop with file:line
2. Names **every trust boundary** crossed and the validation at each
3. Lists **every invariant** the endpoint relies on (cross-ref invariants.md)
4. States what an attacker controls at each hop

**Pass criteria (all required):**
- Zero `???` / "unclear" / "probably" left in the narration
- Every hop has a real file:line, not a hand-wave
- Every trust boundary names its actual (not assumed) validation
- ≥ 3 invariants listed per endpoint, each linked to invariants.md

If the narration contains an unknown → comprehension is incomplete for that path. Loop back
to the relevant subsystem subagent, resolve it, then re-attempt. **Do not proceed to Hunter
with unknowns in the teach-back.**

## SERVICE MODE — targeted comprehension on demand (re-entrant; keeps G9 from dead-ending)

A later stage (Hunter, Tracer, Variant Scanner, Chain Strategist, or the Final Boss triager)
may discover it needs a subsystem that wasn't in the original scope — a sibling outside the
intersection, a chain-link in code Mapper under-weighted. Instead of being blocked by G9 or
hunting blind, that stage calls Analyst in SERVICE MODE:

**Contract:**
- Input: one subsystem name (+ the files/entry points that motivated the request).
- Action: run STEP 2 (single subsystem subagent, Sonnet, bounded ~25 files) + the relevant
  slice of STEP 3 (add its invariants to `invariants.md`, update `domain-model`/`trust-boundaries`
  if affected) + a focused teach-back for the one new entry path.
- Output: `comprehension/subsystem/<name>.md` created; `coverage.md` + master index updated to
  mark it `✅ Understood`; the caller is unblocked.
- Bound: a service call comprehends ONE subsystem. It does not re-run the whole stage. Log the
  call in `§ Feedback Queue` with the requesting stage.
- Single-writer still holds: the service subagent writes its own subsystem doc; the Analyst
  service merges into shared artifacts.

This is the ONLY sanctioned way to expand comprehension after the main stage. It is not a G9
bypass — it satisfies G9 for the new subsystem.

---

## RESUMABILITY (per-stage sub-state)

`comprehension/coverage.md` IS Analyst's resumable sub-state. On resume:
- A subsystem marked `✅ Understood` → skip (doc + passing teach-back already exist).
- A subsystem marked `🔄 In Progress` or absent → (re)spawn its subagent.
- A subsystem marked `⚠️ Comprehension Blocked` → leave blocked; do not retry blindly.
Never redo a comprehended subsystem; never skip an incomplete one. The boot sequence reads
this table, not just the step-level Engagement Progress row.

---

## RELATION TO BLACKBOX-FIRST (no conflict)

This stage is **whitebox by design** — read everything, understand everything. The
Blackbox-First Rule (G1) governs **POC construction in the Exploiter only**: when you build
the exploit, the *payload and parameters* must be derivable from traffic, and reachability
must hold through a real external entry point. Comprehension informs WHAT is vulnerable and
WHY; Blackbox-First governs HOW you prove it later. Use everything you understand here to
pick targets; just don't smuggle internal-only knowledge into the final POC.

## ANALYST ANTI-PATTERNS (do not do these)

- Do NOT grep for sink names — that is Hunter's confirmation tool, not your discovery method
- Do NOT file vulnerabilities — you build understanding; Hunter forms hypotheses, Tracer proves
- Do NOT deep-read out-of-scope files — respect the scope intersection (but bias to inclusion when unsure)
- Do NOT write "looks vulnerable" — write "relies on invariant X, enforced at Y / not enforced"
- Do NOT claim comprehension without a passing teach-back — the artifact is the proof
- Do NOT let one subsystem subagent run unbounded — split it at the file/session cap
- Do NOT fake understanding of obfuscated code — mark it Blocked and route to researcher
- Do NOT let a subagent write the master index or shared artifacts — single-writer (synthesis merges)
- Do NOT skip security-model.md — the triager is blind without it
